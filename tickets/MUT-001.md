# MUT-001: Generalize the pivot runtime adapter into the package-wide runtime bridge

## Summary

Extract the existing pivot-generated runtime bridge into a shared package-side adapter so repair, pivot, and follow-up ticket flows can call generated runtime tools without creating another Python mutation engine.

## Why

The pivot runtime adapter is already the best existing bridge into generated runtime mutations. Generalizing it is the cleanest way to avoid a second package-side mutation engine.

## Package Boundary

- Build the shared bridge in package code.
- Use it to call generated-runtime tools or template-owned contracts; do not add package-root pseudo-runtime state.

## Do This

1. Extract a shared runtime adapter interface from `generated_tool_runtime.py`.
2. Move pivot to the shared adapter without changing pivot behavior.
3. Define how repair and ticket-follow-up callers will invoke runtime-owned mutation tools through the shared bridge.
4. Document the adapter boundary so later tickets do not reintroduce direct package-side mutation logic.

## Files To Update

- `skills/scafforge-pivot/scripts/generated_tool_runtime.py`
- new shared adapter module under package control
- pivot callers that should move to the shared bridge

## Verify

1. Confirm pivot still routes through the adapter after extraction.
2. Confirm the shared bridge can call the runtime-owned mutation surfaces later tickets need.
3. Confirm no new package-side mutation engine is introduced as part of the extraction.
