# LEG-002: Record migration provenance and validate legacy upgrade flows

## Summary

Record explicit migration provenance and validate migrated repo behavior so later repair passes can distinguish migrated state from untouched legacy state and prove that upgrade paths preserve publish-gate truth.

## Why

Migration is only safe if later repair runs can tell what changed and the proof harness can validate those upgrade paths end to end.

## Package Boundary

- Update package provenance and package validation.
- Do not represent migration state through package-root runtime files.

## Do This

1. Extend migration provenance so later repair runs can identify migrated repos and the migration path taken.
2. Add legacy-upgrade validation to the existing migration validator.
3. Cover publish-gate and restart behavior after migration, not just the migration mechanics.
4. Distinguish untouched legacy repos from migrated repos in later repair analysis.

## Files To Update

- provenance helpers touched by migration
- `scripts/validate_gpttalker_migration.py`
- fixtures or seeders used by migration validation

## Verify

1. Run `python3 scripts/validate_gpttalker_migration.py` after the migration proof changes.
2. Confirm migrated repos are identifiable in provenance.
3. Confirm post-migration repair and restart behavior stays truthful in the validation cases.
